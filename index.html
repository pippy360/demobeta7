The Problem

Reverse image search is a brilliantly useful tool. It is also a frustrating tool. It frequently fails to recognize lightly edited, cropped, skewed or rotated images. It returns images of a similar colour before sections of the same image - which is often what the user is looking for but just as often, it’s not. It’s the sort of tool that shows how far computer science has come and also how far it has to go before it compares to the human eye in terms of image-processing. Seeing a problem that puzzled and intrigued me, I set about trying to fix this little quirk.

Now I can’t claim to have solved this issue, nor did I ever really expect to compete with powerhouses like Google or Microsoft or TinEye. However, what I was able to do was to achieve better results when matching certain types of images using math that even a high school student should be able to understand*. I’m going to walk you through, step-by-step, how I found a real-world application for the subject that everyone seemed to hate.

The steps below reference the input image but it should be noted that all of these steps have been carried out on all the images in our database already. This pre-processing of images allow the image matching to be completed in sublinear time.

In order to find commonalities between two images, we need to be able to identify “shapes” within the image. This is achieved in four steps: firstly, we use canny edge detection [link here], which gives us a series of points that show the “edges” of the image (Figure 1a). Next, we have used OpenCV [link?] to find the contours in the image. In simple terms, this turns our points into continuous lines (Figure 1b). Now we need to turn these continuous lines into shapes - some of these lines form clear shapes already; circles, squares, rectangles, etc. However, some of the contours may not form closed loops so we create shapes out of their end points (Figure 1c). Finally, we remove shapes below a certain area threshold because as the number of pixels gets very low, the likelihood that we return a false positive match increases. We refer to the remaining shapes as “valid shapes” (Figure 1d).

Once we have identified all of the shapes within the image, matching an image in our database to the same, unedited input image is simple. The real question is: how do we find matches for images that have been altered? In order to make the problem more manageable, I focused my efforts on matching images that have had 2D affine transformations applied to them. This means images which have been scaled, skewed, rotated, cropped or subjected to any combination of these transformations.

In technical terms, a 2D affine transformation is any transformation which can be expressed with a 2x2 matrix. Figure 2 below shows how various 2D affine transformations affect the a square. Each row in the matrix represents the x,y co-ordinates of a vertex of the square (the matrix of a pentagon would have five rows, a hexagon six rows, etc.).

We now have to think about each of these different transformations separately. Solving the scaling issue is straightforward; we simply scale the shapes until they have some arbitrary chosen area (say 1,000 pixels.).

Rotation is a considerably more difficult transformation to deal with and I must admit that my solution is neither elegant nor foolproof but it does have that most important of qualities: it works. When the pre-processing of images in the database is done, the algorithm is run on each image 360 times, changing the rotation by 1 degree each time.

Normalizing the shape

For the purposes of this article we call this “normalizing” the shape. Reducing the sum of the squared distance from the centre of the shape to every point on its edge to its local minimum value.

How do we actually do it?

Finding a common scale is easy, we just scale the images until the shapes both have some arbitrarily chosen area (say 1000 pixels).

I haven't come up with a good way of finding a common rotation so for a temporary fix I repeat the algorithm 360 times changing the rotation by 1 degree each time.

Finding a common transformation is more complicated and requires a more in depth explanation and some maths.

Finding a common transformation

Firstly we’re ONLY going to handle images that have had 2d affine transformations applied to them. So we can say for sure that each image will be a 2d affine transformed version of a matching image. Most importantly working with only 2d affine transformations makes the maths doable. So what are 2d affine transformations? they are any transformation which can be expressed with a 2x2 matrix. The cool thing about solving something analytically is that after we do all that maths we're left with a really small simple function. For example our answer in pseudo code is:

//Get the 2 parameters of a transformation matrix [[a,b],[0,1/a]] which will spread our input shape out evenly in all directions
//xs = the sum of the square of all the x values of all our points in our shape
//xy = the sum of the square of all the y values of all our points in our shape
//xy = the sum of the product of all the y*x values of all our points in our shape

function get_a(xs, ys, xy) {
    return sqrt(sqrt( pow(ys ,2)/(xs*ys - pow(xy, 2)) ))
}

function get_b(xs, ys, xy) {
    return get_a(xs, ys, xy) * (-xy/ys)
}
